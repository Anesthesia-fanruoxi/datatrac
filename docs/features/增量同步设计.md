# 增量同步设计（CDC 方案）

## 📋 文档说明

本文档详细描述 DataTrace 系统基于 MySQL Binlog 的 CDC（Change Data Capture）增量同步机制。

**文档版本**: v3.0  
**最后更新**: 2026-02-27  
**技术栈**: Go + Gin + GORM + Redis + go-mysql  
**实现状态**: 设计阶段

---

## 🎯 设计目标

### 核心需求

1. **实时数据同步**: 捕获源库的实时变更（INSERT/UPDATE/DELETE）
2. **数据零丢失**: 全量+增量无缝衔接，保证数据完整性
3. **断点续传**: 支持中断后从上次位置继续
4. **高性能**: 异步处理，不影响源库性能
5. **可观测**: 实时监控延迟、速度、错误

### 技术选型

**Binlog 解析**：
- `go-mysql` - 纯 Go 实现，性能好，社区活跃

**消息队列**：
- **Redis Stream**（推荐）- 持久化、消费组、轻量级
- **内存 Channel** - 简单快速，但重启丢失
- **本地文件队列** - 无外部依赖，但性能一般

## 🔄 数据零丢失方案

### 核心问题

```
T0: 开始同步，记录 Binlog 位置 (file=mysql-bin.000001, pos=1000)
   ↓
T1-T5: 全量同步进行中（可能需要几小时）
   ├─ 期间源库持续有新数据写入
   ├─ Binlog 继续增长 (pos=1000 → 5000)
   └─ 这些增量数据如何处理？❓
   ↓
T6: 全量同步完成
   ↓
T7: 开始增量同步
   └─ 如何保证 T1-T6 期间的数据不丢失？❓
```

### 解决方案：全量+增量并行

```
┌─────────────────────────────────────────┐
│ 阶段1: 记录起始位置（快照点）            │
│ - 获取当前 Binlog 位置                   │
│ - 开启事务（保证一致性）                 │
│ - Snapshot Point: file=xxx, pos=1000    │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 阶段2: 并行执行                          │
│                                         │
│  [全量同步线程]        [增量监听线程]     │
│       ↓                      ↓          │
│  读取存量数据          监听 Binlog       │
│  (T0 时刻的快照)       (从 T0 开始)      │
│       ↓                      ↓          │
│  写入目标库            写入 Redis Stream │
│                       (暂不应用)         │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│ 阶段3: 全量完成，切换增量                 │
│ - 全量同步完成                           │
│ - 开始消费 Redis Stream                  │
│ - 应用 T0 之后的所有变更                  │
└─────────────────────────────────────────┘
```

### 一致性保证

#### 使用事务快照

```sql
-- 1. 开启一致性快照
START TRANSACTION WITH CONSISTENT SNAPSHOT;

-- 2. 获取 Binlog 位置
SHOW MASTER STATUS;
-- 结果: file=mysql-bin.000001, position=1000

-- 3. 执行全量查询
SELECT * FROM table1;
SELECT * FROM table2;

-- 4. 提交事务
COMMIT;
```

**优点**：保证全量数据和 Binlog 位置的一致性

### 核心代码流程

```go
// 启动增量同步
func (s *IncrementalSync) Start() error {
    // 1. 记录快照点
    snapshot, err := s.captureSnapshot()
    if err != nil {
        return err
    }
    
    // 2. 启动 Binlog 监听（异步）
    queue := s.createQueue() // Redis Stream 或 Channel
    go s.startBinlogListener(snapshot, queue)
    
    // 3. 执行全量同步
    err = s.fullSync(snapshot)
    if err != nil {
        return err
    }
    
    // 4. 标记全量完成
    s.markFullSyncCompleted()
    
    // 5. 启动增量消费
    go s.consumeIncrementalEvents(queue)
    
    return nil
}

// 捕获快照点
func (s *IncrementalSync) captureSnapshot() (*Snapshot, error) {
    // 开启事务
    tx, _ := s.sourceDB.Begin()
    defer tx.Rollback()
    
    // 获取 Binlog 位置
    var file string
    var pos uint32
    tx.QueryRow("SHOW MASTER STATUS").Scan(&file, &pos, ...)
    
    return &Snapshot{
        File:      file,
        Position:  pos,
        Timestamp: time.Now(),
    }, nil
}

// Binlog 监听器
func (s *IncrementalSync) startBinlogListener(snapshot *Snapshot, queue BinlogQueue) {
    syncer := replication.NewBinlogSyncer(...)
    syncer.StartSync(mysql.Position{
        Name: snapshot.File,
        Pos:  snapshot.Position,
    })
    
    for {
        event, _ := syncer.GetEvent()
        binlogEvent := s.parseEvent(event)
        if binlogEvent != nil {
            queue.Push(binlogEvent) // 写入队列
        }
    }
}

// 增量消费
func (s *IncrementalSync) consumeIncrementalEvents(queue BinlogQueue) {
    for {
        event, _ := queue.Pop()
        
        // 应用变更
        err := s.applyEvent(event)
        if err != nil {
            log.Error(err)
            continue
        }
        
        // ACK 确认
        queue.Ack(event.ID)
        
        // 更新位置
        s.updatePosition(event.BinlogFile, event.BinlogPos)
    }
}

// 应用事件
func (s *IncrementalSync) applyEvent(event *BinlogEvent) error {
    switch event.Type {
    case "INSERT":
        return s.handleInsert(event)
    case "UPDATE":
        return s.handleUpdate(event)
    case "DELETE":
        return s.handleDelete(event)
    }
    return nil
}
```

## 🎯 队列方案对比

### 方案 1：内存队列（Channel）

**优点**：
- ✅ 无需外部依赖
- ✅ 性能最好
- ✅ 实现简单

**缺点**：
- ❌ 重启丢失数据
- ❌ 无法断点续传
- ❌ 不适合长时间全量同步

**适用场景**：
- 开发测试环境
- 数据量小，全量同步很快（几分钟内）

### 方案 2：Redis Stream（推荐）

**优点**：
- ✅ 数据持久化（AOF/RDB）
- ✅ 支持断点续传
- ✅ 消费组支持多消费者
- ✅ ACK 机制保证至少一次消费
- ✅ 轻量级，易部署

**缺点**：
- ❌ 需要部署 Redis
- ❌ 单机性能有限

**适用场景**：
- 生产环境
- 数据量大，全量同步时间长
- 需要高可靠性

### 方案 3：本地文件队列

**优点**：
- ✅ 无需外部依赖
- ✅ 数据持久化

**缺点**：
- ❌ 性能一般
- ❌ 无法多消费者
- ❌ 文件管理复杂

**适用场景**：
- 无法部署 Redis 的环境
- 单机部署

## 📈 监控指标

### 关键指标

1. **增量延迟**：
   - 当前 Binlog 位置 - 源库最新位置
   - 单位：秒

2. **消费速度**：
   - 每秒处理的事件数
   - 单位：events/s

3. **队列积压**：
   - 队列中未消费的事件数
   - 单位：条

4. **错误率**：
   - 应用失败的事件比例
   - 单位：%

### 监控实现

```go
// 增量同步监控
type IncrementalMonitor struct {
    TaskID            string
    CurrentPos        uint32
    LatestPos         uint32
    Lag               int64  // 延迟（秒）
    EventsPerSecond   int64  // 消费速度
    QueueSize         int64  // 队列积压
    ErrorRate         float64 // 错误率
}

// 计算延迟
func (m *IncrementalMonitor) CalculateLag() int64 {
    // 通过 Binlog 位置差异估算延迟
    posDiff := m.LatestPos - m.CurrentPos
    // 假设每秒产生 1000 个事件
    return int64(posDiff / 1000)
}
```
