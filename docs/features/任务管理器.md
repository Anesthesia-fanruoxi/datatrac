# 任务管理器 (TaskManager)

## 📋 文档说明

本文档详细描述 DataTrace 系统的任务管理器实现，包括架构、核心服务、执行流程和关键机制。

**文档版本**: v3.0  
**最后更新**: 2026-03-03  
**技术栈**: Go + Gin + GORM + SSE

---

## 🎯 概述

任务管理器是 DataTrace 系统的核心模块，负责管理数据同步任务的完整生命周期，从创建、配置、执行到监控。

**核心职责**:
- 📦 任务生命周期管理（创建、配置、启动、暂停、停止）
- 🔄 并发控制（Worker池 + 外键专用Worker）
- 📊 实时进度监控和日志推送（SSE）
- 💾 数据库持久化（任务配置、运行状态、单元进度）
- 🔗 外键依赖分析和处理

## 📦 数据模型

### 三表结构

#### 1. sync_tasks - 任务主表
存储任务的基本信息和配置

**关键字段**:
- `id`: 任务ID
- `name`: 任务名称
- `source_id` / `target_id`: 数据源ID
- `config`: JSON配置（包含数据库选择、表映射、同步参数）
- `status`: 任务状态（idle/configured/running）
- `is_running`: 是否正在运行
- `current_step`: 当前步骤（initialize/sync_data/validate）
- `sync_mode`: 同步模式（full/incremental）

#### 2. task_unit_configs - 任务单元配置表
存储任务包含的所有同步单元（表）

**关键字段**:
- `id`: 单元配置ID
- `task_id`: 所属任务ID
- `unit_name`: 单元名称（格式：目标库.目标表）
- `unit_type`: 单元类型（table）

#### 3. task_unit_runtimes - 任务单元运行时表
存储任务单元的运行状态和进度

**关键字段**:
- `id`: 运行时ID
- `task_id`: 所属任务ID
- `unit_name`: 单元名称
- `status`: 状态（pending/running/completed/failed/paused）
- `total_records`: 总记录数
- `processed_records`: 已处理记录数
- `error_message`: 错误信息
- `started_at`: 开始时间
- `updated_at`: 更新时间

### 任务配置结构（JSON）

存储在 `sync_tasks.config` 字段中：

```
{
  "source_id": "数据源ID",
  "target_id": "目标数据源ID",
  "selected_databases": [
    {
      "database": "目标数据库名",
      "source_database": "源数据库名",
      "is_database_modified": false,
      "tables": [
        {
          "source_table": "源表名",
          "target_table": "目标表名",
          "is_modified": false
        }
      ]
    }
  ],
  "sync_config": {
    "batch_size": 1000,
    "thread_count": 4,
    "sync_mode": "full",
    "error_strategy": "skip",
    "table_exists_strategy": "drop"
  }
}
```

## 🔧 核心服务

### 1. TaskService - 任务管理服务

**职责**: 任务的CRUD操作和配置管理

**核心功能**:
- **创建任务**: 创建新的同步任务，初始状态为 idle
- **配置任务**: 设置数据源、选择数据库和表、配置同步参数
- **生成任务单元**: 根据配置自动生成 task_unit_configs 记录
- **查询任务**: 获取任务列表和详情
- **删除任务**: 级联删除任务及其所有单元配置和运行记录

**关键逻辑**:
- 配置验证：检查数据源类型匹配、不能自己同步自己
- 任务单元生成：遍历所有选中的表，生成 `目标库.目标表` 格式的单元名称
- 状态管理：配置完成后将状态从 idle 更新为 configured

---

### 2. TaskControlService - 任务控制服务（核心调度器）

**职责**: 任务的启动、暂停、停止控制和执行调度

**核心功能**:

#### 2.1 启动任务 (StartTask)

**执行流程**:

```
1. 前置检查
   ├─ 验证任务状态（必须是 configured）
   ├─ 检查是否已在运行
   └─ 如果所有单元已完成，重置为 pending

2. 初始化任务单元运行记录
   ├─ 检查 task_unit_runtimes 是否存在
   ├─ 不存在则从 task_unit_configs 创建
   └─ 状态设为 pending

3. 外键分析阶段
   ├─ 按数据库分组任务单元
   ├─ 连接源数据库查询外键关系
   ├─ 使用拓扑排序得到删除/创建顺序
   └─ 标记哪些表有外键依赖

4. 更新任务状态
   ├─ is_running = true
   └─ current_step = "initialize"

5. 创建执行上下文
   ├─ 创建 context.Context（支持取消）
   ├─ 创建 TaskExecution 对象
   └─ 存储到 executions map

6. 启动异步执行
   └─ 启动 goroutine 执行两阶段流程
```

**两阶段执行流程**:

```
阶段1: 初始化（initialize）
  ├─ 创建所有数据库（从源库获取字符集）
  ├─ 处理表结构
  │   ├─ drop策略：先删除所有表（子→父），再创建所有表（父→子）
  │   ├─ truncate策略：清空所有表
  │   └─ append策略：检查并创建不存在的表
  └─ 更新每个单元的 total_records

阶段2: 数据同步（sync_data）
  ├─ 分离外键表和普通表
  ├─ 反转外键表顺序（变成插入顺序：父→子）
  ├─ 创建两个队列：fkQueue 和 normalQueue
  ├─ 启动外键Worker（workerID=-1，串行处理外键表）
  ├─ 启动普通Worker池（并发处理普通表）
  └─ 等待所有Worker完成
```

#### 2.2 暂停任务 (PauseTask)

**执行流程**:
1. 验证任务正在运行
2. 调用 context.Cancel() 发送取消信号
3. 等待所有Worker退出（WaitGroup.Wait）
4. 更新任务状态：is_running = false
5. 更新运行中的单元状态为 paused

#### 2.3 停止任务 (StopTask)

**执行流程**:
1. 发送取消信号并等待Worker退出
2. 更新任务状态：is_running = false
3. 重置所有单元状态为 pending
4. 清空进度数据（processed_records = 0）

**关键机制**:
- **执行管理**: 使用 sync.Map 存储所有正在执行的任务
- **取消机制**: 使用 context.Context 实现优雅停止
- **并发控制**: 使用 WaitGroup 等待所有Worker完成

---

### 3. SyncEngine - 同步引擎

**职责**: 执行具体的数据同步逻辑

**核心功能**:

#### 3.1 InitializeWorker - 初始化Worker

**职责**: 串行处理所有表的初始化（创建数据库和表结构）

**执行流程**:
1. 按数据库分组任务单元
2. 连接源数据库获取字符集和排序规则
3. 在目标库创建数据库
4. 根据策略处理表结构：
   - drop: 分两阶段（删除 + 创建）
   - truncate/append: 直接处理
5. 更新每个单元的 total_records

#### 3.2 Worker - 数据同步Worker

**职责**: 从队列中获取任务单元并执行数据同步

**执行流程**:
1. 从 channel 获取任务单元
2. 更新状态为 running
3. 创建 Reader 和 Writer
4. 批量读取和写入数据
5. 每批次更新进度并推送
6. 完成后标记为 completed

**Worker类型**:
- **外键Worker** (workerID=-1): 串行处理外键表，保证顺序
- **普通Worker** (workerID=0,1,2...): 并发处理普通表

#### 3.3 SyncTable - 同步单个表

**职责**: 执行单个表的数据同步（表结构已在初始化阶段处理）

**执行流程**:
1. 解析表名和配置
2. 创建 Reader 和 Writer
3. 检查是否为空表
4. 批量读取数据（按 batch_size）
5. 批量写入数据
6. 更新进度（内存 + 数据库 + SSE）
7. 处理错误（根据 error_strategy）

---

### 4. ForeignKeyAnalyzer - 外键分析器

**职责**: 分析表的外键依赖关系并进行拓扑排序

**核心功能**:

#### 4.1 分析外键关系

**查询逻辑**:
- 从源库的 `information_schema.KEY_COLUMN_USAGE` 查询外键
- 只保留子表在同步列表中的外键关系
- 构建外键关系列表：`[(子表, 父表), ...]`

#### 4.2 拓扑排序（Kahn算法）

**排序逻辑**:
1. 构建有向图：parent → child
2. 计算每个节点的入度
3. 从入度为0的节点开始BFS遍历
4. 得到创建顺序（父表在前）
5. 反转得到删除顺序（子表在前）

**返回结果**:
- 排序后的表列表（删除顺序）
- 是否有外键依赖（布尔值）
- 外键表集合（用于分离外键表和普通表）

---

### 5. TaskProgressService - 进度聚合服务

**职责**: 聚合任务的整体进度信息

**核心功能**:

#### 5.1 获取任务进度

**聚合逻辑**:
1. 查询所有任务单元的运行时记录
2. 统计各状态的单元数量
3. 计算总进度百分比
4. 计算处理速度（记录数/秒）
5. 估算剩余时间
6. 获取当前步骤信息

**返回数据**:
- 总单元数 / 已完成数 / 运行中数 / 失败数
- 总记录数 / 已处理记录数
- 整体进度百分比
- 处理速度
- 预计剩余时间
- 当前步骤（initialize/sync_data/validate）
- 正在运行的表列表
- 失败的表列表

---

### 6. TaskLogService - 日志管理服务

**职责**: 管理任务执行过程中的日志

**核心功能**:
- **日志存储**: 使用内存 map 存储日志（taskID → []TaskLog）
- **日志分类**: info/success/warning/error
- **日志类别**: all/create/sync/complete
- **日志限制**: 每个任务最多保留1000条日志
- **控制台输出**: 同时输出到控制台便于调试

**日志格式**:
```
{
  "time": "15:04:05",
  "level": "info",
  "message": "表 db.table 批次 1: 1000/10000 (10.0%)",
  "category": "sync"
}
```

---

### 7. TaskSSEService - SSE推送服务

**职责**: 通过Server-Sent Events向前端推送实时更新

**核心功能**:

#### 7.1 客户端管理
- 使用 map 存储每个任务的订阅客户端
- 客户端断开时自动清理
- 支持多个客户端同时订阅同一任务

#### 7.2 消息推送
- **进度更新**: 推送任务整体进度
- **日志更新**: 推送新增的日志条目
- **状态变更**: 推送任务状态变化

**推送时机**:
- 每批次数据处理完成后
- 任务单元状态变化时
- 任务启动/暂停/停止时

**消息格式**:
```
{
  "type": "progress" | "log" | "status",
  "data": { ... }
}
```

## 📊 关键机制

### 1. 外键依赖处理机制

#### 问题背景
- 删除表时遇到外键约束：Error 3730
- 插入数据时遇到外键约束：Error 1452
- 并发同步时，子表数据插入时父表数据可能还不存在

#### 解决方案

**三个关键顺序**:

1. **删除顺序**（子表 → 父表）
```
qrtz_simprop_triggers (子表)
qrtz_cron_triggers (子表)
qrtz_simple_triggers (子表)
qrtz_blob_triggers (子表)
qrtz_triggers (父表)
qrtz_job_details (最顶层父表)
```

2. **创建顺序**（父表 → 子表）
```
qrtz_job_details (最顶层父表)
qrtz_triggers (父表)
qrtz_blob_triggers (子表)
qrtz_simple_triggers (子表)
qrtz_cron_triggers (子表)
qrtz_simprop_triggers (子表)
```

3. **数据同步顺序**（父表 → 子表）
```
与创建顺序相同
```

#### 实现方式

**外键分析**:
- 查询源库的 `information_schema.KEY_COLUMN_USAGE`
- 只保留子表在同步列表中的外键关系
- 使用拓扑排序（Kahn算法）得到正确顺序

**混合并发模式**:
- 外键表：使用专用Worker串行处理，保证顺序
- 普通表：使用Worker池并发处理，提高速度
- 两者并行执行，互不阻塞

---

### 2. 并发控制机制

#### Worker池模式

**设计原理**:
- 使用 channel 作为任务队列
- 使用 goroutine 作为Worker
- 使用 WaitGroup 等待所有Worker完成
- 使用 context.Context 实现取消机制

**Worker类型**:
- **外键Worker**: workerID=-1，串行处理外键表
- **普通Worker**: workerID=0,1,2,3...，并发处理普通表

**并发数控制**:
- 由用户配置 `thread_count` 参数
- 默认值为4
- 外键Worker不占用并发数配额

#### 取消机制

**实现方式**:
- 使用 `context.WithCancel` 创建可取消的上下文
- 每个Worker在处理前检查 `ctx.Done()`
- 暂停/停止时调用 `cancel()` 发送取消信号
- 使用 `WaitGroup.Wait()` 等待所有Worker优雅退出

---

### 3. 进度推送机制

#### 推送架构

```
SyncEngine (数据同步)
    ↓ 每批次完成
TaskProgressService (进度聚合)
    ↓ 计算整体进度
TaskSSEService (SSE推送)
    ↓ 推送消息
前端 (实时显示)
```

#### 推送内容

**进度信息**:
- 总单元数 / 已完成数
- 总记录数 / 已处理记录数
- 整体进度百分比
- 处理速度（记录数/秒）
- 预计剩余时间
- 当前步骤（initialize/sync_data）
- 正在运行的表列表
- 失败的表列表

**日志信息**:
- 时间戳
- 日志级别（info/success/warning/error）
- 日志消息
- 日志分类（all/create/sync/complete）

#### 推送时机
- 每批次数据处理完成后
- 任务单元状态变化时
- 任务启动/暂停/停止时

---

### 4. 错误处理机制

#### 错误策略

**skip策略**（跳过）:
- 记录错误日志
- 标记当前单元为 failed
- 继续处理其他单元
- 适用于非关键数据

**pause策略**（暂停）:
- 记录错误日志
- 标记当前单元为 failed
- 停止所有Worker
- 保存断点，支持恢复
- 适用于关键数据

#### 常见错误处理

**Error 1390**（占位符超限）:
- 自动分批插入
- 动态计算每批次的记录数
- 确保占位符数量不超过65535

**Error 3730**（外键约束-删除）:
- 按外键依赖顺序删除表
- 先删除子表，再删除父表

**Error 1452**（外键约束-插入）:
- 使用外键Worker串行处理
- 按父表→子表顺序插入数据

---

### 5. 断点续传机制

#### 实现原理

**状态持久化**:
- 每个任务单元的状态存储在 `task_unit_runtimes` 表
- 记录已处理的记录数（processed_records）
- 记录单元状态（pending/running/completed/failed/paused）

**恢复逻辑**:
1. 启动任务时检查是否有未完成的单元
2. 过滤出 status != completed 的单元
3. 从上次的 processed_records 继续处理
4. 使用 LIMIT + OFFSET 跳过已处理的数据

**重新启动**:
- 如果所有单元都已完成，重置所有单元状态为 pending
- 清空 processed_records
- 重新开始全量同步

---

### 6. 数据库连接池优化

#### 问题背景
- 每个Worker的Reader和Writer都有独立的连接池
- 并发数过高时，连接数可能超过数据库限制

#### 优化方案

**连接池参数**:
- `MaxOpenConns`: 2（每个Reader/Writer最多2个连接）
- `MaxIdleConns`: 1（最多1个空闲连接）
- `ConnMaxLifetime`: 1分钟（连接1分钟后回收）
- `ConnMaxIdleTime`: 30秒（空闲30秒后关闭）

**效果**:
- 4个Worker × 2个连接（Reader+Writer）× 2个最大连接 = 16个连接
- 远低于MySQL默认的151个连接限制

## 💡 使用场景

### 场景1: 创建并配置任务

**流程**:
1. 调用 `POST /api/tasks` 创建任务
2. 调用 `PUT /api/tasks/:id/config` 配置任务
   - 选择源和目标数据源
   - 选择要同步的数据库和表
   - 配置同步参数（批量大小、线程数、策略等）
3. 系统自动生成任务单元配置
4. 任务状态变为 configured，可以启动

### 场景2: 启动全量同步

**流程**:
1. 调用 `POST /api/tasks/:id/start` 启动任务
2. 系统执行外键分析
3. 初始化阶段：创建数据库和表结构
4. 数据同步阶段：混合并发模式同步数据
5. 前端通过SSE实时显示进度和日志

### 场景3: 监控任务进度

**流程**:
1. 前端订阅 `GET /api/tasks/:id/sse`
2. 系统推送进度更新消息
3. 前端实时更新进度条和表格
4. 显示当前步骤、处理速度、剩余时间

### 场景4: 处理任务失败

**流程**:
1. 任务执行过程中某个表同步失败
2. 根据错误策略：
   - skip: 记录错误，继续其他表
   - pause: 暂停任务，保存断点
3. 用户查看错误日志
4. 修复问题后重新启动任务
5. 系统从断点继续执行

### 场景5: 暂停和恢复

**流程**:
1. 调用 `POST /api/tasks/:id/pause` 暂停任务
2. 系统发送取消信号，等待Worker退出
3. 保存当前进度到数据库
4. 调用 `POST /api/tasks/:id/start` 恢复任务
5. 系统从上次的进度继续执行

## 🎯 设计优势

### 1. 分阶段处理
- 初始化和数据同步分离
- 逻辑清晰，易于维护
- 便于添加新阶段（如数据校验）

### 2. 混合并发模式
- 外键表串行处理，保证正确性
- 普通表并发处理，提高性能
- 两者并行执行，充分利用资源

### 3. 实时反馈
- SSE推送实时进度和日志
- 用户体验好
- 便于监控和调试

### 4. 断点续传
- 状态持久化到数据库
- 支持暂停和恢复
- 失败后可以重试

### 5. 灵活配置
- 支持多种表存在策略（drop/truncate/append）
- 支持多种错误策略（skip/pause）
- 可配置批量大小和线程数

### 6. 外键智能处理
- 自动分析外键依赖
- 自动排序和分组
- 避免外键约束错误

### 7. 连接池优化
- 精细控制连接数
- 避免连接数过多
- 自动回收空闲连接

## 📈 性能优化

### 1. 批量处理
- 使用批量读取和批量写入
- 减少网络往返次数
- 默认批量大小1000，可配置

### 2. 并发控制
- 使用Worker池并发处理
- 默认4个Worker，可配置
- 充分利用多核CPU

### 3. 连接复用
- Reader和Writer复用数据库连接
- 减少连接建立开销
- 优化连接池参数

### 4. 进度推送优化
- 每批次推送一次，避免频繁推送
- 使用SSE而非轮询，减少服务器压力
- 前端自动重连机制

### 5. 内存优化
- 批量处理，不一次性加载所有数据
- 及时释放资源
- 使用流式处理
