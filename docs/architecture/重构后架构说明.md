# 重构后架构说明

> 更新时间：2024年
> 
> 本文档说明了去掉进度表和配置表后的新架构设计

## 📊 架构变更概述

### 变更原因

1. **简化架构** - 进度表和配置表增加了系统复杂度
2. **提升性能** - 全量同步不需要频繁写数据库
3. **明确职责** - 配置、进度、统计分别存储在最合适的地方

### 核心变更

- ❌ 删除 `task_unit_configs` 表
- ❌ 删除 `task_unit_runtimes` 表
- ✅ 配置详情存储到 Redis
- ✅ 运行时进度维护在内存
- ✅ 增量统计存储到 Redis

---

## 🏗️ 数据存储架构

### 1. MySQL - 基础配置和最终状态

**存储内容**：
- 数据源信息（`data_sources`）
- 任务基本信息（`sync_tasks`）
- 任务最终状态（`is_running`, `current_step`）

**表结构**：

```sql
-- 数据源表
CREATE TABLE `data_sources` (
    `id` VARCHAR(36) PRIMARY KEY,
    `name` VARCHAR(100) NOT NULL,
    `type` VARCHAR(20) NOT NULL,
    `host` VARCHAR(255) NOT NULL,
    `port` INT NOT NULL,
    `username` VARCHAR(100) NOT NULL,
    `password` VARCHAR(255) NOT NULL,
    `database_name` VARCHAR(100),
    `created_at` TIMESTAMP,
    `updated_at` TIMESTAMP
);

-- 同步任务表
CREATE TABLE `sync_tasks` (
    `id` VARCHAR(36) PRIMARY KEY,
    `name` VARCHAR(100) NOT NULL,
    `source_id` VARCHAR(36),
    `target_id` VARCHAR(36),
    `source_type` VARCHAR(20) NOT NULL,
    `target_type` VARCHAR(20) NOT NULL,
    `config` TEXT NOT NULL,              -- JSON格式的基本配置
    `status` VARCHAR(20) NOT NULL,       -- idle/configured
    `is_running` BOOLEAN DEFAULT FALSE,  -- 是否正在运行
    `sync_mode` VARCHAR(20) NOT NULL,    -- full/incremental
    `current_step` VARCHAR(50),          -- initialize/sync_data/completed/incremental
    `queue_type` VARCHAR(20),            -- memory/redis
    `created_at` TIMESTAMP,
    `updated_at` TIMESTAMP
);
```

### 3. 内存 - 运行时进度

**存储内容**：

```go
type TaskProgressData struct {
    TaskID      string
    Units       map[string]*TaskUnit  // unitName -> unit
    StartTime   time.Time
    CurrentStep string
}

type TaskUnit struct {
    UnitName         string
    Status           string    // pending/running/completed/failed/initialized
    TotalRecords     int64
    ProcessedRecords int64
    ErrorMessage     string
    StartedAt        *time.Time
    UpdatedAt        time.Time
}
```

**特点**：
- 任务启动时初始化
- 任务完成后清空
- 不持久化到数据库

## 🔄 同步模式对比

### 全量同步模式

```
启动
  ↓
从 Redis 读取配置（task:config:{taskID}）
  ↓
初始化内存进度（TaskProgressManager）
  ↓
外键分析和排序
  ↓
阶段1：初始化（创建数据库和表结构）
  ├─ 更新内存状态：initialized
  └─ SSE 推送进度
  ↓
阶段2：数据同步（多Worker并发）
  ├─ 更新内存进度：running → completed
  └─ SSE 推送进度
  ↓
完成
  ├─ 更新 MySQL：is_running=false, current_step=completed
  ├─ 清空内存进度
  └─ SSE 推送最终状态
```

**数据流**：
- 配置：Redis → 内存
- 进度：内存 → SSE → 前端
- 状态：内存 → MySQL（仅最终状态）

---

### 增量同步模式

```
启动
  ↓
捕获 Binlog 快照点
  ↓
┌─────────────────────┬─────────────────────┐
│   线程1：Binlog监听  │   线程2：全量同步    │
│                     │                     │
│  监听 Binlog 事件    │  从 Redis 读取配置   │
│       ↓             │       ↓             │
│  写入 Redis 队列     │  初始化内存进度      │
│  (持续后台运行)      │       ↓             │
│                     │  阶段1：初始化       │
│                     │       ↓             │
│                     │  阶段2：数据同步     │
│                     │       ↓             │
│                     │  SSE 推送进度        │
└─────────────────────┴─────────────────────┘
              ↓
        全量同步完成
              ↓
    更新 MySQL：current_step=incremental
              ↓
        启动增量消费
              ↓
    从 Redis 队列读取 Binlog 事件
              ↓
    应用到目标库
              ↓
    更新 Redis 统计（task:stats:{taskID}）
              ↓
    SSE 推送统计信息
```

**数据流**：
- 配置：Redis → 内存
- 全量进度：内存 → SSE → 前端
- Binlog事件：Binlog → Redis队列
- 增量统计：Redis → SSE → 前端
- 状态：内存 → MySQL（仅最终状态）

## 🔧 服务启动流程

```
1. 加载配置文件（config.yaml）
   ↓
2. 初始化 MySQL 连接
   ├─ 自动迁移表结构（data_sources, sync_tasks）
   └─ 不再创建 task_unit_configs 和 task_unit_runtimes
   ↓
3. 初始化 Redis 连接
   ↓
4. 加载所有已配置任务到 Redis
   ├─ 查询 MySQL：status='configured'
   ├─ 解析 config 字段
   └─ 写入 Redis：task:config:{taskID}
   ↓
5. 启动 HTTP 服务
```

## 🎯 配置修改流程

```
前端提交配置
  ↓
API 接收请求
  ↓
验证配置
  ↓
更新 MySQL
  ├─ sync_tasks.config（JSON）
  ├─ sync_tasks.status = 'configured'
  └─ sync_tasks.current_step = ''
  ↓
删除 Redis 旧配置
  └─ DEL task:config:{taskID}
  ↓
重新加载到 Redis
  └─ SET task:config:{taskID} {配置JSON}
  ↓
返回成功
```

## 📈 进度查询流程

### 全量同步进度查询

```
前端请求 /api/tasks/{id}/progress
  ↓
查询 MySQL 获取任务基本信息
  ├─ sync_mode
  └─ current_step
  ↓
从内存读取进度（TaskProgressManager）
  ├─ 统计各状态表数
  ├─ 计算总体进度
  ├─ 计算同步速度
  └─ 估算剩余时间
  ↓
返回 JSON
```

---

### 增量同步进度查询

```
前端请求 /api/tasks/{id}/progress
  ↓
查询 MySQL 获取任务基本信息
  ├─ sync_mode = 'incremental'
  └─ current_step = 'incremental'
  ↓
从 Redis 读取统计（task:stats:{taskID}）
  └─ 每个表的 insert/update/delete 数量
  ↓
返回 JSON
```

---

## 🗑️ 任务删除流程

```
前端请求删除任务
  ↓
检查任务状态
  └─ is_running = true → 拒绝删除
  ↓
清理 Redis 配置
  └─ DEL task:config:{taskID}
  ↓
清理 Redis 统计
  └─ DEL task:stats:{taskID}
  ↓
清理内存进度
  └─ TaskProgressManager.ClearTask(taskID)
  ↓
删除 MySQL 记录
  └─ DELETE FROM sync_tasks WHERE id = ?
  ↓
返回成功
```

## 💡 设计优势

### 1. 性能提升
- 全量同步无数据库IO，完全在内存
- 减少了数据库写入压力
- 提高了同步速度

### 2. 架构简化
- 删除了2个表，减少了表关联
- 代码更清晰，逻辑更简单
- 维护成本降低

### 3. 职责明确
- MySQL：持久化核心配置
- Redis：缓存详情配置和统计
- 内存：临时进度数据

### 4. 灵活扩展
- Redis配置可以支持更复杂的映射关系
- 内存进度结构可以灵活调整
- 不受数据库表结构限制

---

## ⚠️ 注意事项

### 1. Redis 依赖
- 系统依赖 Redis 存储配置
- Redis 故障时可以从 MySQL 降级读取基本配置
- 建议配置 Redis 持久化（RDB + AOF）

### 2. 内存管理
- 任务完成后及时清理内存进度
- 避免内存泄漏
- 大量并发任务时注意内存使用

### 3. 数据一致性
- 配置修改时先更新 MySQL，再更新 Redis
- 服务重启时从 MySQL 重建 Redis 配置
- 增量统计数据仅在 Redis，不持久化到 MySQL

## 🔄 迁移指南

### 从旧版本升级

1. **备份数据**
   ```sql
   -- 备份旧表数据（如需要）
   CREATE TABLE task_unit_configs_backup AS SELECT * FROM task_unit_configs;
   CREATE TABLE task_unit_runtimes_backup AS SELECT * FROM task_unit_runtimes;
   ```

2. **删除旧表**
   ```sql
   DROP TABLE IF EXISTS task_unit_runtimes;
   DROP TABLE IF EXISTS task_unit_configs;
   ```

3. **重启服务**
   - 服务启动时会自动加载配置到 Redis

4. **验证**
   - 检查 Redis 中是否有配置：`KEYS task:config:*`
   - 测试任务启动和进度查询
