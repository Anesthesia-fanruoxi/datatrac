# 全量同步流程优化

## 优化概述

本次优化将全量同步流程从"逐表处理"模式重构为"分阶段处理"模式，并增加了外键依赖分析和处理机制。

## 优化前的问题

### 1. 流程问题
- **逐表处理**：每个表都要经历"创建数据库 → 处理表结构 → 同步数据"三个步骤
- **重复操作**：同一个数据库会被重复创建多次（虽然有判断，但仍有开销）
- **外键冲突**：删除表时遇到外键约束错误
- **数据插入失败**：并发同步时，子表数据插入时父表数据可能还不存在

### 2. 性能问题
- 逐表串行处理，无法充分利用并发
- 外键表和普通表混在一起处理，效率低

### 3. 可维护性问题
- 初始化和同步逻辑耦合在一起
- 代码复杂，难以理解和维护

## 优化后的方案

### 核心思想
**分阶段处理 + 外键依赖分析 + 混合并发模式**

### 执行流程

```
┌─────────────────────────────────────────────────────────────┐
│                     启动任务                                  │
└─────────────────────┬───────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────────┐
│              外键分析阶段（查询源库）                          │
│  - 查询所有表的外键关系                                        │
│  - 只保留子表在同步列表中的外键                                 │
│  - 拓扑排序得到删除/创建顺序                                   │
└─────────────────────┬───────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────────┐
│          阶段1: 初始化（串行，按外键顺序）                      │
│                                                               │
│  步骤1: 创建所有数据库                                         │
│    - 从源库获取字符集和排序规则                                 │
│    - 在目标库创建数据库                                        │
│                                                               │
│  步骤2: 删除所有表（子表 → 父表）                              │
│    - 按外键依赖的删除顺序                                      │
│    - 先删除子表，再删除父表                                    │
│                                                               │
│  步骤3: 创建所有表（父表 → 子表）                              │
│    - 按外键依赖的创建顺序（删除顺序的反转）                     │
│    - 先创建父表，再创建子表                                    │
│    - 使用 SHOW CREATE TABLE 获取完整表结构（包含外键）         │
└─────────────────────┬───────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────────────────┐
│          阶段2: 数据同步（混合并发模式）                        │
│                                                               │
│  ┌─────────────────────┐      ┌──────────────────────────┐  │
│  │  外键Worker         │      │  普通Worker池             │  │

│  │                     │      │                          │  │
│  │  串行处理外键表      │      │  并发处理普通表           │  │
│  │  （父表 → 子表）     │      │  （无顺序要求）           │  │
│  │                     │      │                          │  │
│  │  qrtz_job_details   │      │  sys_user                │  │
│  │  qrtz_triggers      │      │  sys_role                │  │
│  │  qrtz_blob_triggers │      │  sys_menu                │  │
│  │  ...                │      │  ...                     │  │
│  └─────────┬───────────┘      └──────────┬───────────────┘  │
│            │                             │                  │
│            └──────────┬──────────────────┘                  │
│                       ↓                                     │
│              所有表同步完成                                   │
└─────────────────────────────────────────────────────────────┘
```

## 技术实现

### 1. 外键分析（`services/foreign_key_analyzer.go`）

#### 查询外键关系
```sql
SELECT 
    kcu.TABLE_NAME as child_table,
    kcu.REFERENCED_TABLE_NAME as parent_table
FROM information_schema.KEY_COLUMN_USAGE kcu
WHERE kcu.TABLE_SCHEMA = ?
    AND kcu.REFERENCED_TABLE_NAME IS NOT NULL
    AND kcu.TABLE_NAME != kcu.REFERENCED_TABLE_NAME
ORDER BY kcu.TABLE_NAME, kcu.CONSTRAINT_NAME
```

#### 过滤规则
- 只保留子表在同步列表中的外键关系
- 父表可以不在同步列表中（因为外键约束仍然存在）

#### 拓扑排序（Kahn算法）
- 构建有向图：parent → child
- 入度为0的是父表（没有依赖）
- BFS遍历得到创建顺序
- 反转得到删除顺序

### 2. 初始化阶段（`services/sync_init.go`）

#### InitializeWorker 方法
```go
func (e *SyncEngine) InitializeWorker(ctx context.Context, taskID string, units []*models.TaskUnitRuntime) error
```

**职责**：
1. 创建所有数据库
2. 根据策略处理表结构

**drop 策略的特殊处理**：
- 阶段1：按删除顺序删除所有表
- 阶段2：按创建顺序（删除顺序的反转）创建所有表

**truncate/append 策略**：
- 直接按顺序处理即可

#### dropTable 方法
```go
func (e *SyncEngine) dropTable(ctx context.Context, taskID string, unit *models.TaskUnitRuntime, ...) error
```

**职责**：只删除表，不创建

#### createTable 方法
```go
func (e *SyncEngine) createTable(ctx context.Context, taskID string, unit *models.TaskUnitRuntime, ...) error
```

**职责**：
1. 从源库获取表结构（使用 SHOW CREATE TABLE）
2. 在目标库创建表（包含外键约束）
3. 更新总记录数

### 3. 数据同步阶段（`services/task_control_service.go`）

#### 分离表
```go
var fkUnits []*models.TaskUnitRuntime    // 有外键依赖的表
var normalUnits []*models.TaskUnitRuntime // 无外键依赖的表
```

#### 反转外键表顺序
- `sortedUnits` 是删除顺序（子表在前，父表在后）
- 数据同步需要插入顺序（父表在前，子表在后）
- 对 `fkUnits` 进行反转

#### 混合并发模式
```go
// 外键Worker（串行）
go func() {
    engine.Worker(ctx, taskID, fkQueue, -1)
}()

// 普通Worker池（并发）
for i := 0; i < threadCount; i++ {
    go func(workerID int) {
        engine.Worker(ctx, taskID, normalQueue, workerID)
    }(i)
}
```

### 4. 同步引擎（`services/sync_engine.go`）

#### SyncTable 方法简化
- 移除数据库创建逻辑
- 移除表结构处理逻辑
- 只负责数据读取和写入

### 5. 进度展示（`services/task_progress_service.go` + 前端）

#### 新增字段
- `CurrentStep`：当前步骤（initialize/sync_data/validate）

#### 前端展示
```javascript
const stepNames = {
    'initialize': '初始化（创建数据库和表结构）',
    'sync_data': '数据同步',
    'validate': '数据校验'
};
```

## 性能对比

### 优化前
```
时间 = Σ(每个表的处理时间)
     = Σ(创建数据库 + 处理表结构 + 同步数据)
```

### 优化后
```
时间 = 初始化时间 + max(外键表同步时间, 普通表同步时间)
     = 创建所有数据库 + 处理所有表结构 + max(外键表, 普通表)
```

### 性能提升
1. **数据库创建**：从 N 次减少到 M 次（M 是数据库数量）
2. **表结构处理**：集中处理，减少连接开销
3. **数据同步**：外键表和普通表并行，充分利用并发

**示例**：
- 假设有 484 个表，其中 6 个有外键
- 4 个Worker线程

**优化前**：
- 484 个表串行或混乱并发处理
- 可能遇到外键冲突，需要重试

**优化后**：
- 初始化：串行处理 484 个表（但只是结构，很快）
- 数据同步：
  - 外键Worker：串行处理 6 个表
  - 普通Worker池：4个Worker并发处理 478 个表
- 总时间 ≈ 初始化时间 + max(6个表的时间, 478个表/4的时间)

## 关键优化点

### 1. 所有信息从源库获取
- 数据库字符集和排序规则
- 表结构（包括索引、外键）
- 数据记录

**原则**：目标库只是执行操作的地方，不依赖目标库的任何信息

### 2. 外键处理的三个关键顺序

#### 删除顺序（子表 → 父表）
```
qrtz_simprop_triggers (子表)
qrtz_cron_triggers (子表)
qrtz_simple_triggers (子表)
qrtz_blob_triggers (子表)
qrtz_triggers (父表)
qrtz_job_details (最顶层父表)
```

#### 创建顺序（父表 → 子表）
```
qrtz_job_details (最顶层父表)
qrtz_triggers (父表)
qrtz_blob_triggers (子表)
qrtz_simple_triggers (子表)
qrtz_cron_triggers (子表)
qrtz_simprop_triggers (子表)
```

#### 数据同步顺序（父表 → 子表）
```
与创建顺序相同
```

### 3. 混合并发模式
- **外键表**：串行处理，保证顺序
- **普通表**：并发处理，提高速度
- **两者并行**：最大化资源利用

### 4. 错误处理优化
- **Error 1390**：自动分批插入，避免占位符超限
- **Error 3730**：按正确顺序删除表，避免外键约束冲突
- **Error 1452**：按正确顺序插入数据，避免外键约束失败

## 总结

本次优化通过分阶段处理和混合并发模式，在保证正确性的前提下，大幅提升了全量同步的性能和可维护性。核心改进包括：

1. **正确性**：完善的外键依赖分析和处理机制
2. **性能**：混合并发模式，充分利用资源
3. **可维护性**：清晰的阶段划分，代码结构更合理
4. **可观测性**：步骤信息展示，进度更清晰

这为后续的增量同步和数据校验功能奠定了良好的基础。
